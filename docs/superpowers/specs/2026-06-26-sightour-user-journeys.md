## 0. 文档元信息

### 用途
本文档定义 Sightour 用户从「决定来中国」到「离开中国」全过程中的 5 个最关键任务。每个任务分解为 5–7 个阶段，每个阶段标注：用户动作、应用接触点、思考/感受、痛点、超预期机会点。

这是产品决策与设计决策的中观桥梁——把画像层（who）和屏级决策层（what）连接起来。

### 读者
- 产品经理（功能优先级仲裁）
- UX 设计师（屏级流程编排）
- 内容运营（决定每个阶段需要什么文案/POI 数据）
- 工程师（识别性能 / 离线 / 国际化关键路径）

### 与其他文档的关系

- **下游**：信息架构（IA）、屏级流程图、状态设计（loading/empty/error）、微文案指南
- **平行**：[PRD §3 分模块详细功能需求](file:///c:/Users/wenmo/.trae-cn/worktrees/旅游app/Sightour.MD) — 本文档不重复功能 spec，专注于用户视角的「时刻 + 感受 + 机会」

### 信息源声明
基于 2026-06-26 用户画像文档中三类画像的行程叙事，结合中国入境 / 上海旅行实况的公开知识构建。

### 5 个 top task 总览

| # | Journey | 跨周期 | 涉及画像 | 核心价值 |
| - | --- | --- | --- | --- |
| 1 | 行前准备 | 行前 7–30 天 → 登机口 | 全部 3 类 | 第一次信任建立 |
| 2 | 落地转换 | 着陆 → 第一晚入住 | 全部 3 类 | 物理与数字身份落地 |
| 3 | 日常寻找 | 中期每天循环 | 全部 3 类 | 反复使用习惯养成 |
| 4 | 危机处理 | 不定期突发事件 | 全部 3 类 | 信任深化或彻底流失 |
| 5 | 离境收尾 | 最后一天 → 离境后 30 天 | Emma / James | 复购与口碑来源 |

### 每阶段 5 维度说明
- **动作（Action）**：用户在做什么（点击、说话、移动）
- **接触点（Touchpoint）**：用户在用 APP 的哪个屏 / 哪个 UI 元素
- **思考 / 感受（Thought / Feeling）**：用户的认知状态和情绪
- **痛点（Pain）**：用户的摩擦、沮丧、不确定
- **机会（Opportunity）**：APP 可以超出预期的具体机会

## 1. Journey · 行前准备

**场景**：用户在出发前 7–30 天决定「装这个 APP」并完成首次配置，最终在登机口完成心理与数字准备。

**涉及画像**：Emma（商务，决策链短）、James & Maria（自由行，研究型）、Lin Wei（留学，行前已被学校强制引导）

**关键成功指标**：从「搜索签证」到「登机口离线包可用」的总耗时 < 20 分钟；首次配置后 30 天内回访率 ≥ 60%。

### 阶段 1.1 — 签证资格自查

| 维度 | 内容 |
| --- | --- |
| **动作** | 用户在 Google / Reddit / Tripadvisor 搜索「China visa for [nationality] 2026」，看到 Sightour 推荐或广告，点进应用商店。 |
| **接触点** | 商店页面 / 应用首屏（onboarding 第 1 屏） |
| **思考 / 感受** | 焦虑（不知道是否需要签证、过期了没有、政策变没变）。Lin Wei 已确认有 X1 签证，所以此阶段相对松弛。 |
| **痛点** | 政策分散在国家移民管理局官网（中文）+ 当地中国领事馆页面（多语言）+ Reddit 帖子（过期）。Sightour 若不直接给出明确结论，用户会去查 5 个来源。 |
| **机会** | ① 第 1 屏就是一个国家选择器，立刻给答案 ② 答案后接「官方政策原文链接」增加可信度 ③ 直接说「Your passport / 30-day visa-free / No advance application needed」这种结构化结论，不写散文。 |

### 阶段 1.2 — APP 安装与首次进入

| 维度 | 内容 |
| --- | --- |
| **动作** | 用户下载 APP，启动后看到 onboarding 第 1 屏（品牌 + 3 句价值主张），点「Get started」进入语言选择。 |
| **接触点** | Onboarding 第 1 屏（已存在）→ 第 2 屏语言选择 |
| **思考 / 感受** | 判断「这 APP 看起来正不正经」。Lin Wei 已被学校 PDF 介绍过，会跳过这一步。 |
| **痛点** | 第 1 屏若文字过多或动画冗长，Emma / James 会直接关掉。 |
| **机会** | ① 第 1 屏 3 句话内必须传达价值主张 ② 文案直接说「No booking · No commission · Just info」破除预期防线 ③ 默认中文/英文双语切换在首屏可见，消除「我不确定能不能用」的疑虑。 |

### 阶段 1.3 — 选择语言与离线包下载

| 维度 | 内容 |
| --- | --- |
| **动作** | 用户选语言 → 进入 Home → 滚到「Filter by nationality」→ 选自己国家 → 看到 3 张政策卡 + 1 张「Tourist visa guide」→ 滚下去到「Pre-arrival checklist」→ 开始勾选项。 |
| **接触点** | 政策卡 3 张 + 清单（已存在）；离线包下载入口在 Tools → Offline downloads（已存在） |
| **思考 / 感受** | 「我现在搞清楚了吗？」决定此人是会每天用还是下飞机就删 APP。 |
| **痛点** | ① 政策卡如果都是中文，用户立刻放弃 ② 离线包如果不显眼，到了中国没数据时才发现 ③ 清单如果不显示进度，用户觉得「做完没做完不清楚」。 |
| **机会** | ① 政策卡顶部加按国籍筛选项（已实现） ② 进入 Home 时若检测到未下载离线包，主动推送系统级通知「Download offline pack before departure?」 ③ 清单进度条始终可见（已实现，3/7）。 |

### 阶段 1.4 — 机场准备（行前 12 小时）

| 维度 | 内容 |
| --- | --- |
| **动作** | Emma / James 装好行李准备出门。Lin Wei 由家人陪同去机场。最后一公里——打车去机场。 |
| **接触点** | 可能打开 APP 看汇率、紧急号码；不太会再探索新功能。 |
| **思考 / 感受** | 兴奋（要出发了）+ 焦虑（落地的想象）。Emma 时间压力最大，可能在出租车上快速扫一眼 APP 确认机场交通。 |
| **痛点** | ① 若用户此刻才打开 APP，没下载离线包则飞 11 小时后落地无数据 ② 紧急号码如果在 APP 深处而非首屏，落地第一晚手机没信号时找不到。 |
| **机会** | ① 离线包下载完成后给系统级「行前清单全部完成 ✓」推送 ② 紧急号码（110/120/119 + 32 国领事馆）做成可「添加到主屏」快捷方式（iOS / Android 都支持），让用户落地无网络也能一键拨打。 |

### 阶段 1.5 — 飞行中

| 维度 | 内容 |
| --- | --- |
| **动作** | 用户在飞机上想「落地第一步做什么」。长飞行（Emma 11h、James 14h 转机）= 大量空闲。 |
| **接触点** | APP 已下载但可能没网络（机上 WiFi 贵）。离线包起作用。 |
| **思考 / 感受** | 从「OK 装好了」转向「落地那 90 分钟会发生什么？」 |
| **痛点** | ① 若离线包没下载，看不到任何内容 ② 即便下了，飞行中 WiFi 贵，用户可能不想联网。 |
| **机会** | ① 离线包覆盖「机场到酒店」的所有可能路径 ② 飞行模式下默认打开首屏是「I just landed」清单 ③ 文案主动告诉用户「接下来 90 分钟你可以这样操作」减少焦虑。 |

## 2. Journey · 落地转换

**场景**：从飞机降落出关到第一晚入住。这是三画像中**压力最高的 90 分钟**。

**涉及画像**：Emma（追求高效、不出错）、James & Maria（首次来华、迷路恐惧）、Lin Wei（必须跑多个政府窗口）

**关键成功指标**：从「下飞机」到「入住」耗时 Emma < 60 min / James < 90 min / Lin Wei 第 1 天 < 3 hr；第 1 天内回访 APP ≥ 3 次。

### 阶段 2.1 — 着陆与出关

| 维度 | 内容 |
| --- | --- |
| **动作** | 用户下飞机 → 跟着标识走 → 出境检查 → 提取行李 → 进入到达大厅。 |
| **接触点** | 手机开机 → APP 仍在打开状态或再次打开 |
| **思考 / 感受** | 「终于到了」「接下来怎么办？」Lin Wei 第一时间找大学联络员举的牌子。 |
| **痛点** | ① 出关队伍长，用户无聊时会刷 APP ② 出关后人群拥挤，手机信号弱（地下通道尤甚）。 |
| **机会** | ① APP 提供「机场到市区」分步教程（地铁/出租车/网约车三种选项）② Lin Wei 触发「我是学生」分支，给出大学接机人位置与确认方式。 |

### 阶段 2.2 — 机场到市区

| 维度 | 内容 |
| --- | --- |
| **动作** | Emma：核对酒店接送单（中文，不会读）。James：Booking.com 接送司机没来，要临时打车。Lin Wei：找学校联络员。 |
| **接触点** | 紧急联系人屏 + 出行工具屏 + 常用语（医疗 / 交通） |
| **思考 / 感受** | Emma 焦虑（公司报销的单子要核对）；James 慌（迷之自信的英文在中国的出租车上失效）；Lin Wei 紧张（被陌生大叔叫名字）。 |
| **痛点** | ① Emma 的接送单是中文 ② James 的司机失联 ③ Lin Wei 可能被骗（联络员假冒）。 |
| **机会** | ① APP 给 Emma 显示「Show driver this hotel name in Chinese」预制卡片（含中文 + 拼音 + 英文） ② James 切到 DiDi 国际版下单前，APP 提示「Show driver the destination in Chinese: 外滩瑞吉酒店」 ③ Lin Wei 联络员识别卡：让用户先扫码大学官方联络员工牌，无工牌不跟。 |

### 阶段 2.3 — 入住酒店 / 宿舍

| 维度 | 内容 |
| --- | --- |
| **动作** | Emma / James 在酒店前台办理入住。Lin Wei 已经在宿舍。 |
| **接触点** | 入住不需要 APP，但 Lin Wei 需在 APP 看「24 小时 police registration」清单。 |
| **思考 / 感受** | Emma 终于松一口气；James 担心前台会不会要护照照片、押金等额外文件；Lin Wei 焦虑「我必须在 24h 内去派出所」。 |
| **痛点** | ① James 不知道押金怎么退 ② Lin Wei 不知道登记要带什么、几点去、能不能代办。 |
| **机会** | ① Hotel check-in 流程短卡片（中英对照「退房时间 / 早餐 / WiFi 密码」）② Lin Wei 的「24h 登记」清单：所需文件、办公时间、最近派出所地址、是否接受英文邮件预约。 |

### 阶段 2.4 — SIM 卡 / 移动网络

| 维度 | 内容 |
| --- | --- |
| **动作** | Emma 用德国漫游（贵且流量少）。James 找机场便利店买 eSIM 或实体 SIM。Lin Wei 已在线买好 eSIM，落地激活。 |
| **接触点** | Tools → Currency converter（看 ¥300 套餐值不值） |
| **思考 / 感受** | James 第一次被中国 SIM 流程搞晕（实名制、激活要上传护照照片）。Lin Wei 已经提前搞定，相对轻松。 |
| **痛点** | ① James 不知道机场哪些运营商支持外籍护照 ② 激活流程纯中文 ③ 没有 eSIM 经验的 James 买错卡。 |
| **机会** | ① APP 推荐 3 家支持外籍护照的运营商（按 James 国籍过滤） ② 提供「Top-up guide」中文操作截图英文化版本 ③ 离线包已经包含离线汇率换算，避免落地后流量不够还查汇率。 |

### 阶段 2.5 — 第一晚回顾

| 维度 | 内容 |
| --- | --- |
| **动作** | 用户躺在床上刷 APP，看看明天能去哪。 |
| **接触点** | Home（行前清单）→ Discover / Map → POI 列表 |
| **思考 / 感受** | Emma 在整理第二天议程；James 在兴奋地刷地图；Lin Wei 在列第二天政府窗口行程。 |
| **痛点** | ① Emma 第二天需要浦东附近商务场所 ② James 想去「非游客陷阱」餐厅 ③ Lin Wei 不知道哪些餐厅不会让她遇到猪肉。 |
| **机会** | ① Emma 看到「Business meeting venues near Pudong」分类（如果 P1 上线）② James 用 Discover 看 YouTube 上看到的「本地点」，APP 标记「本地热门 + 外宾友好」 ③ Lin Wei 找清真餐厅（如果 P1 上线）。 |

## 3. Journey · 日常寻找

**场景**：用户在中期每天循环使用 APP 寻找地点、交通、餐厅、景点。这是**决定 DAU / WAU** 的旅程。

**涉及画像**：全部 3 类（频次：Emma 高频短查 / James 高频研究型 / Lin Wei 中频刚需型）

**关键成功指标**：每日打开次数 ≥ 2；平均会话时长 30s–3min；POI 详情点击率 ≥ 40%。

### 阶段 3.1 — 决定「今天做什么」

| 维度 | 内容 |
| --- | --- |
| **动作** | Emma 早晨看日程，决定午休 1h 找附近的咖啡馆安静开会。James 看天气决定去室内。Lin Wei 周末决定找一家清真餐厅。 |
| **接触点** | Home / Discover / Map（顶导搜索框） |
| **思考 / 感受** | Emma 模式化操作（每天都用）；James 模式未定型（首日后尚未形成习惯）；Lin Wei 任务明确但搜索质量差。 |
| **痛点** | ① Emma 不知道附近哪支持外卡 ② James 担心「游客陷阱」 ③ Lin Wei 找清真难。 |
| **机会** | ① Emma 看到外卡友好筛选 ② James 用 Discover 榜单（避坑 + 热门） ③ Lin Wei 等 P1 清真过滤功能。 |

### 阶段 3.2 — 搜索 POI

| 维度 | 内容 |
| --- | --- |
| **动作** | 用户在 Map 屏搜索关键词，或在 POI 列表里浏览。 |
| **接触点** | Map 搜索框 / 筛选 chip / POI 列表卡片 / 5 维评分 / 「本店 vs 品类均值」条形图（均已实现） |
| **思考 / 感受** | James 高度依赖可视化对比；Emma 快速扫一眼评分就走；Lin Wei 需要看「外宾友好」+「支付便利」两个维度。 |
| **痛点** | ① POI 卡片信息太多（80+ 字段），用户不知道看哪个 ② 地图标记和列表不同步 ③ 搜索结果过少。 |
| **机会** | ① POI 卡片首屏只显示：评分 + 2 个标签 + 距离 + 价格档 ② 详情页按 persona 智能排序维度（Emma 看商务便利度，James 看地道度，Lin Wei 看支付便利度 + 清真） ③ 搜索时若输入英文，自动尝试匹配中文（反向 geocoding）。 |

### 阶段 3.3 — 评估可信度

| 维度 | 内容 |
| --- | --- |
| **动作** | 用户在 POI 详情页查看 5 维评分 + 第三方引用 + 提示。 |
| **接触点** | POI Detail 屏（雷达图 + 横向柱形图已实现） |
| **思考 / 感受** | 「这是不是游客陷阱？James 此刻最关心这个。 |
| **痛点** | ① 评分体系如果太复杂（5 个维度太多），用户懒得看 ② 没有说明评分的来源与权威性。 |
| **机会** | ① 详情页首屏突出「本店 vs 品类均值」对比图（已实现），让用户在 5 秒内做判断 ② 显示「Verified · No sponsorships」徽章（已实现） ③ 「Heads up」tip 卡（已实现）单独区分「这家店的雷区」。 |

### 阶段 3.4 — 抵达

| 维度 | 内容 |
| --- | --- |
| **动作** | 用户点 POI 详情页的「Directions」按钮，调起系统地图导航。 |
| **接触点** | POI Detail → Directions 按钮 + 「Sightour 不预订、不售卖 — 此按钮仅打开您手机的地图或拨号」声明（已实现） |
| **思考 / 感受** | 期望 APP 能告诉我「到那儿后该做什么」。 |
| **痛点** | ① Directions 只调起地图，没告诉我「到了之后怎么进门」「在哪一层」「哪个入口近」 ② 营业时间 / 排队预估缺失。 |
| **机会** | ① POI 详情页加「Arrival checklist」3-5 条：找到大堂吧 → 报名字预约 → 电梯到 5F → 出示预订单 ② 实时排队预估（如果 P2 接入商家 API）。 |

### 阶段 3.5 — 支付与体验

| 维度 | 内容 |
| --- | --- |
| **动作** | 用户在 POI 完成消费。James 刷卡失败，留现金。Emma 刷卡成功。Lin Wei 用支付宝。 |
| **接触点** | POI 卡片的 Chip（支持外卡 / 仅现金 / 支付宝国际版 等） |
| **思考 / 感受** | James 痛点爆发：「又被拒了」；Emma 流畅但会想：「明天还不知道会遇到什么」。 |
| **痛点** | ① Chip 信息滞后（POI 标明「支持外卡」但今天机器坏了） ② 没有 fallback 方案 ③ 不确定要不要给小费。 |
| **机会** | ① POI 详情页加「Payment tips」卡：本店支持外卡 / 需 PIN / 现金兜底 ② 加小费指南（中式服务默认不含小费） ③ 显示「最近用户反馈」（P2 时由用户主动提交）。 |

## 4. Journey · 危机处理

**场景**：突发事件——生病 / 丢护照 / 被骗 / 紧急事故。这是**信任的放大器或粉碎机**。

**涉及画像**：全部 3 类（James 生病概率最高 / Emma 行程变更高 / Lin Wei 居留风险最高）

**关键成功指标**：危机事件下 APP 启用率 ≥ 80%；危机事件 7 天后回访率 ≥ 50%（信任留存）。

### 阶段 4.1 — 危机发生

| 维度 | 内容 |
| --- | --- |
| **动作** | James 醒来胃不舒服；Emma 客户改时间酒店需重订；Lin Wei 发现护照丢了。 |

## 附录 B · 跨 Journey 的关键洞察

### B.1 离线能力是最高优先级
5 个 Journey 中，**4 个的关键路径**依赖离线能力（行前下载 / 落地无信号 / 危机无信号 / 离境退税）。如果离线包不完整，整个产品的核心价值主张崩塌。

**建议**：MVP 离线包必须包含——
- 全部 POI 数据
- 全部常用语（含发音）
- 紧急号码
- 5 维评分
- 离线地图瓦片（至少上海核心区）
- 离线汇率换算（最近一次同步数据）

### B.2 「第一晚」是真实留存率的决定点
Lin Wei 与 James & Maria 的行程都明示：「第 7 天起，他们要么形成使用习惯，要么彻底放弃」。也就是说**前 7 天的体验**决定 LTV。

**建议**：前 7 天每日推送「今日 tip」型非侵入通知（如「今天的天气适合逛外滩」「你还没去过任何一家博物馆，要不要看 3 个选项？」）。不推送会让用户忘记 APP，推送过多则卸载。

### B.3 危机 Journey 是「流失放大器」
普通使用失败用户最多沉默；但危机 Journey 失败（如紧急号码打不通、药店信息过期）会让用户**永久负面推荐**。

**建议**：危机 Journey 的 SLA 应当比常规 Journey 严格 10 倍。所有紧急内容必须每月人工核验；医院 / 领事馆 / 紧急号码必须有自动监控告警。

### B.4 跨旅程的 3 个共同接触点
1. **首屏的「Filter by nationality」**——在 5 个 Journey 中都被使用，应作为 APP 的「身份锚点」
2. **Tools 屏**——常用语 + 紧急号码 + 汇率，行前/落地/危机/离境都用
3. **POI 详情页**——日常/危机/离境都依赖，是单点投入产出最高的屏

### B.5 「不在 APP 内做」的场景
为了再次确认 Phase 1 边界：
- 不在 APP 内订机票 / 酒店 / 餐厅
- 不在 APP 内办签证 / 居留许可
- 不在 APP 内叫车 / 叫外卖（DiDi 国际版 / 美团国际版）
- 不在 APP 内社交 / 评价

APP 的存在价值是「**让用户在做这些事之前能更聪明地决策**」。

## 附录 D · 待后续讨论的开放问题

1. **「离线包大小」预算**——Lin Wei 4 年学生可能装 4 个城市的离线包。James 一次旅行只需要 1 个。需要多大？影响功能边界。
2. **「推送」的产品边界**——危机触发推送很敏感（侵犯隐私 vs 救命功能）。需要单独 design review。
3. **「Last day in Shanghai」清单**——是否上线前必须有？还是 MVP 后由数据决定？
4. **「政策变更推送」频次**——如果用户停留 7 天内遇政策变化（如某国临时暂停免签），APP 是否要推送？频次上限？
5. **「第三方分享卡」生成**——生成什么样的视觉卡？能否生成深链接到 POI 详情？
